# Bus Named Outputs & Display Names — Design Spec

**Date:** 2026-04-13
**Ticket:** ByteBard97/SignalCanvasLang#9
**Status:** Approved

---

## Problem

Two data-loss bugs in the PatchLang bus round-trip, blocking the BNE-MTG project:

1. **Named outputs are dropped.** `InternalBusOutput.name` (e.g. "Link 1-L") has no representation in PatchLang syntax. The emitter flattens all named outputs into bare `output: Port` lines, silently discarding every output name.

2. **Bus display names are not emitted.** `BusEntry.label: Option<String>` already exists in the AST and the parser reads `label: "..."`, but the formatter never emits it. Bus display names containing special characters (e.g. `"PQ>MM"`) are lost on round-trip.

---

## Domain Context

In a mixing console or router, every bus output is a named signal — "Link 1-L", "IEM Mix 1", "Matrix Out → Lobby". Engineers name these because they need to know what they're sending. An unnamed bus output is not a meaningful concept in the domain.

The frontend type `InternalBusOutput.name: string` is non-optional — the UI always has a name for every output. The old `output: Port` syntax was a bug artifact from the broken emitter, not an intentional language feature. Since PatchLang has not been deployed to production users, no backward compatibility constraint applies.

---

## Design Decisions

### Decision 1: Labels are required on bus outputs

Output labels are mandatory. There is no unlabeled output syntax. Empty string labels (`output "": Port`) are rejected at parse time. This matches the frontend domain model and eliminates the silent data loss class permanently.

### Decision 2: Unrouted outputs are supported

An output can be declared with a name but no destination — representing a mix that exists in the bus manager but hasn't been cabled yet. This is a real production state (bus outputs reserved before the MADI patch is finalised).

### Decision 3: Multi-destination outputs are supported

`InternalBusOutput.destinations` is a Vec — one named output can route to multiple ports (e.g. a mix sent to both MADI and Dante simultaneously). PatchLang must represent this without data loss. Syntax uses comma-separated port refs.

### Decision 4: Old `output: Port` syntax is removed

No backward compatibility shim. Files are machine-generated by the emitter; the emitter will be updated. Existing `.patch` files (MTG.patch, TestSave.patch) will round-trip correctly after the emitter update.

### Decision 5: Inputs remain `Vec<PortRef>`

`InternalBusInput` has no name field — inputs are bare channel references in the frontend model. Keeping `inputs: Vec<PortRef>` is semantically correct, not an asymmetry to fix.

### Decision 6: Bus display name stays as body-form `label:` key

The existing `label: "..."` syntax inside the bus block body is retained and the formatter is fixed to emit it. The bus identifier line does not change. This is consistent with how `label:` works in `config` blocks and avoids a grammar change to `bus-entry`.

### Decision 7: Duplicate output labels within a bus are a DRC warning

Two outputs in the same bus sharing a label string (e.g. two `output "Main"` entries) is a DRC warning, not a parse error. The frontend disambiguates by ID, but duplicate names are almost certainly a user error in a hand-authored file.

## Syntax

### Bus output (Gap 1)

```
bus Link_1 {
  input: Fader[1..8]
  output "Link 1-L": MADI_1_Out[1]              # labeled, single destination
  output "Link 1-R": MADI_1_Out[2], Dante[5]    # labeled, multiple destinations
  output "Link 1-C"                              # labeled, unrouted
}
```

Grammar delta:

```ebnf
# OLD
bus-port-entry = ( "input" | "output" | "in" | "out" ) ":" port-ref-or-local ;

# NEW
bus-port-entry   = bus-input-entry | bus-output-entry ;
bus-input-entry  = ( "input" | "in" ) ":" port-ref-or-local ;
bus-output-entry = ( "output" | "out" ) string-literal
                   [ ":" port-ref-or-local { "," port-ref-or-local } ] ;
```

The quoted label is required and must be non-empty. The `:` and port reference list are optional — omitting both declares an unrouted output. Multiple destinations are comma-separated.

### Bus display name (Gap 2)

The `label:` key inside the bus body is retained. The formatter is fixed to emit it:

```
bus PQMM {
  label: "PQ>MM"
  input: Fader[1]
  output "Main": Matrix_Out[1]
}
```

The bus identifier line (`bus PQMM`) does not change. This is a formatter-only fix — no grammar or parser change required for Gap 2.

## AST Changes

### New struct

```rust
#[derive(Debug, Clone, Serialize, Deserialize, TS)]
#[ts(export)]
pub struct BusOutput {
    pub label: String,              // required, non-empty
    pub destinations: Vec<PortRef>, // empty = unrouted; 1+ = routed
    pub span: Span,
}
```

### `BusEntry` change

```rust
pub struct BusEntry {
    pub name: String,
    pub label: Option<String>,      // bus display name — unchanged
    pub inputs: Vec<PortRef>,       // unchanged
    pub outputs: Vec<BusOutput>,    // CHANGED from Vec<PortRef>
    pub span: Span,
}
```

## JSON / Compat Layer

### New compat struct

```rust
#[derive(Debug, Clone, Serialize, Deserialize, TS)]
#[ts(export)]
pub struct TsBusOutput {
    pub label: String,
    pub destinations: Vec<TsPortRef>,
}
```

### `TsBusDecl` change

```rust
pub struct TsBusDecl {
    pub name: String,
    pub label: Option<String>,
    pub inputs: Vec<TsPortRef>,
    pub outputs: Vec<TsBusOutput>,  // CHANGED from Vec<TsPortRef>
}
```

### JSON shape

Before:
```json
{
  "outputs": [{ "instance": "", "port": "MADI_1_Out", "indexSpec": [...] }]
}
```

After:
```json
{
  "outputs": [
    {
      "label": "Link 1-L",
      "destinations": [
        { "instance": "", "port": "MADI_1_Out", "indexSpec": [{ "type": "single", "value": 1 }] }
      ]
    }
  ]
}
```

Unrouted output:
```json
{ "label": "Link 1-C", "destinations": [] }
```

## DRC

### Existing rule changes

**S05** (bus references unknown port): The output validation loop changes to iterate `BusOutput.destinations` instead of the flat `outputs: Vec<PortRef>`. An unrouted output (`destinations: []`) skips S05. The input validation loop (`inputs: Vec<PortRef>`) is **unchanged**.

### New rule

**C-new** (duplicate bus output label): A DRC warning when two outputs in the same bus share the same label string. Severity: warning.

*Note: `InternalBus.notes` has no PatchLang representation. This is a pre-existing known gap, out of scope for this ticket.*

## Test Cases

**Parser:**
- `output "Name": Port` — labeled single-destination parses correctly
- `output "Name": Port1, Port2` — labeled multi-destination parses correctly
- `output "Name"` — labeled unrouted parses correctly
- `output: Port` — parse error (old syntax rejected)
- `output ""` — parse error (empty label rejected)
- Bus with mix of routed, multi-destination, and unrouted outputs parses correctly

**Formatter (assert emitted text, not just round-trip AST):**
- `output "Link 1-L": MADI_1_Out[1]` appears literally in formatter output
- `output "Link 1-C"` appears literally for unrouted output
- `label: "PQ>MM"` appears in bus body when `label` is set
- Formatter output re-parses to identical AST (full round-trip)

**Round-trip fidelity:**
- Bus with labeled outputs + display name round-trips without data loss
- Multi-destination output survives round-trip
- Special chars in labels (`>`, `-`, `:`, spaces) survive round-trip

**DRC:**
- S05 fires on unrecognised port in `destinations`
- S05 skips unrouted outputs
- C-new warning fires on duplicate label within same bus
- S05 input loop still fires on unrecognised input port (regression test)

## Open Questions

None. All resolved:
- Input labels: not needed (`InternalBusInput` has no name field)
- Backward compat: not needed (no deployed users)
- Label required vs optional: required (matches frontend domain model)
- Multi-destination: supported via comma-separated syntax
- Display name syntax: body-form `label:` key, formatter fix only
- Duplicate labels: DRC warning
