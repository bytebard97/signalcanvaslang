# Bus Named Outputs & Display Names — Design Spec

**Date:** 2026-04-13
**Ticket:** ByteBard97/SignalCanvasLang#9
**Status:** Approved

---

## Problem

Two data-loss bugs in the PatchLang bus round-trip, blocking the BNE-MTG project:

1. **Named outputs are dropped.** `InternalBusOutput.name` (e.g. "Link 1-L") has no representation in PatchLang syntax. The emitter flattens all named outputs into bare `output: Port` lines, silently discarding every output name.

2. **Bus display names are not emitted.** `BusEntry.label: Option<String>` already exists in the AST and the parser reads `label: "..."`, but the formatter never emits it. Bus display names containing special characters (e.g. `"PQ>MM"`) are lost on round-trip.

---

## Domain Context

In a mixing console or router, every bus output is a named signal — "Link 1-L", "IEM Mix 1", "Matrix Out → Lobby". Engineers name these because they need to know what they're sending. An unnamed bus output is not a meaningful concept in the domain.

The frontend type `InternalBusOutput.name: string` is non-optional — the UI always has a name for every output. The old `output: Port` syntax was a bug artifact from the broken emitter, not an intentional language feature. Since PatchLang has not been deployed to production users, no backward compatibility constraint applies.

---

## Design Decisions

### Decision 1: Labels are required on bus outputs

Output labels are mandatory. There is no unlabeled output syntax. This matches the frontend domain model and eliminates the silent data loss class permanently.

### Decision 2: Unrouted outputs are supported

An output can be declared with a name but no destination — representing a mix that exists in the bus manager but hasn't been cabled yet. This is a real production state (bus outputs reserved before the MADI patch is finalised).

### Decision 3: Old `output: Port` syntax is removed

No backward compatibility shim. Files are machine-generated by the emitter; the emitter will be updated. Existing `.patch` files (MTG.patch, TestSave.patch) will round-trip correctly after the emitter update.

### Decision 4: Inputs remain `Vec<PortRef>`

`InternalBusInput` has no name field — inputs are bare channel references in the frontend model. Keeping `inputs: Vec<PortRef>` is semantically correct, not an asymmetry to fix.

## Syntax

### Bus output (Gap 1)

```
bus Link_1 {
  input: Fader[1..8]
  output "Link 1-L": MADI_1_Out[1]    # labeled + routed
  output "Link 1-R": MADI_1_Out[2]    # labeled + routed
  output "Link 1-C"                    # labeled, unrouted
}
```

Grammar delta:

```ebnf
# OLD
bus-port-entry = ( "input" | "output" | "in" | "out" ) ":" port-ref-or-local ;

# NEW
bus-port-entry = bus-input-entry | bus-output-entry ;
bus-input-entry  = ( "input" | "in" ) ":" port-ref-or-local ;
bus-output-entry = ( "output" | "out" ) string-literal [ ":" port-ref-or-local ] ;
```

The quoted label is required. The `:` and port reference are optional — omitting both declares an unrouted output.

### Bus display name (Gap 2)

Formatter now emits the `label` field when present:

```
bus PQMM "PQ>MM" {
  input: Fader[1]
  output "Main": Matrix_Out[1]
}
```

This was already valid per the language reference; the formatter was simply not emitting it.

## AST Changes

### New struct

```rust
#[derive(Debug, Clone, Serialize, Deserialize, TS)]
#[ts(export)]
pub struct BusOutput {
    pub label: String,              // required — user-defined name
    pub destinations: Vec<PortRef>, // empty = unrouted
    pub span: Span,
}
```

### `BusEntry` change

```rust
pub struct BusEntry {
    pub name: String,
    pub label: Option<String>,      // bus display name — unchanged
    pub inputs: Vec<PortRef>,       // unchanged
    pub outputs: Vec<BusOutput>,    // CHANGED from Vec<PortRef>
    pub span: Span,
}
```

---

## JSON / Compat Layer

`compat_types.rs` `InstanceBusDecl` output shape changes from:

```json
{
  "outputs": [{ "instance": "", "port": "MADI_1_Out", "indexSpec": [...] }]
}
```

to:

```json
{
  "outputs": [
    {
      "label": "Link 1-L",
      "destinations": [{ "instance": "", "port": "MADI_1_Out", "indexSpec": [...] }]
    }
  ]
}
```

The frontend `loadFromPatchLang.ts` and `emitterBuilder.ts` must be updated to match this new shape.

## DRC

No new DRC rules. Existing S05 (bus references unknown port) continues to apply — it iterates `destinations` instead of the old flat `outputs`. An unrouted output (`destinations: []`) skips S05 validation.

New convention check: **C04** (bus declared with zero outputs) already exists and remains unchanged.

---

## Test Cases

- `output "Name": Port` — labeled routed output parses correctly
- `output "Name"` — labeled unrouted output parses correctly
- `output: Port` — parse error (old syntax rejected)
- Bus with mix of routed and unrouted outputs round-trips correctly
- Bus `label:` field survives round-trip (Gap 2)
- Bus display name with special chars (`"PQ>MM"`) survives round-trip
- DRC S05 fires on unrecognised port in `destinations`
- DRC S05 skips unrouted outputs

## Open Questions

None. Resolved during design:
- Input labels: not needed (`InternalBusInput` has no name field)
- Backward compat: not needed (no deployed users)
- Label required vs optional: required (matches domain, matches frontend type)
